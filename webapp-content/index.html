counter-app

For starters, I created a basic Python application that counts the number of HTTP GET requests that were initiated.

I utilized "Flask" to initialize the web server, and "request" in order to get insights about the HTTP GET requests.

The application has a counter that will track the amount of GET requests (App is stateless).

Jenkins Continuous Integration

The Jenkins Server used in this project physically resides in a homelab in my house.

A Jenkins pipeline resides together within the counter-app repo.

The Pipeline uses Jenkins Poll SCM feature in order to detect changes in the repository.

The Pipeline checks out the repository, builds a docker image of the application, and tags the image with the Pipeline BUILD_NUMBER, in addition to a "latest" tag.

After tagging the docker image, the image is pushed to AWS Elastic Container Registry (ECR).

The last and most important thing the pipeline does is update another repo (mock-app-infra), which contains the application helm chart.

Using the "yq" (Command line YAML processor) via the Jenkins pipeline ensures that the application image tag inside values.yaml is in line with the ECR image tag. The pipeline BUILD_NUMBER is utilized here again.

EKS Deployment Using Terraform

Furthermore, I created a GitHub Action pipeline that utilizes Terraform to provision and configure the required resources to deploy a full-blown EKS cluster.

The tfstate file generated by Terraform is stored in an AWS S3 bucket.

Continuous Delivery using ArgoCD

In order to provide full coverage of the CI/CD, I took a GitOps approach to update the Kubernetes application.

I deployed ArgoCD on the cluster and pointed it towards the mock-app-infra repo, which contains the most updated application chart.

ArgoCD ensures the application is always up to date by polling GitHub (every 3 minutes) to detect changes in the application manifests.

Project Summary

By leveraging a multitude of DevOps tools, I successfully implemented a comprehensive Continuous Integration/Continuous Deployment (CI/CD) pipeline that harnesses the power of GitOps, cloud technologies, Infrastructure as Code (IaC), and Source Code Management (SCM).

This integrated approach allowed for efficient and streamlined software development and deployment processes.

Exposing the Bucket with CloudFront

Next, I wanted to make the web server publicly available,
I created a CloudFront distribution and in order to optimize costs I selected the
appropriate price class based on the geographical distribution of my target audience, primarily in Israel.
Additionally, I configured the behavior settings of the CloudFront distribution to automatically redirect HTTP
requests to HTTPS, ensuring a secure browsing experience for users.

SSL Certificate Creation and Validation

To provide enhanced security, I utilized the AWS Certificate Manager,
I created an SSL certificate in the (N. Virginia) (us-east-1) because it is the requirement by CloudFront,
made sure to also create an "alternate domain name" in the CloudFront distribution,
otherwise the distribution will not be accessible through domain name tomerschwartz.com,
Since my DNS records were hosted on Cloudflare, I needed to complete a DNS validation process. To achieve
this, I added the CNAME record provided by AWS Certificate Manager to my Cloudflare DNS settings. Once the
validation process was successfully completed, I applied the SSL certificate to the CloudFront distribution.
Additionally, I configured a CNAME record in Cloudflare, associating my domain (tomerschwartz.com) with the
CloudFront distribution, in order to ensure visitors could access the website using the desired domain name.

GitHub Actions Pipeline for Automated Deployment

To streamline the deployment process and facilitate seamless updates, I set up a GitHub Actions pipeline.
This automation allowed any changes pushed to the repository to trigger automatic deployment to the S3 bucket.
I created a dedicated IAM user for GitHub to ensure the appropriate access rights.
pipeline, I generated an access key for the IAM user and added the corresponding credentials to the GitHub
This allowed the pipeline to access and update the S3 bucket whenever changes

cryptoApp

Project Repository

cryptoApp is a monolithic web application deployed behind an AWS Application Load Balancer. The goal of cryptoApp is to demonstrate how to deploy a web application behind an AWS Application Load Balancer using these tools.

Key Features and Tools:

  • Web server is written in Golang and runs as a systemd service
  • Utilizes EC2 user data to download the dependencies and start the web server
  • Uses AWS Application Load Balancer (ALB) to distribute traffic to EC2 instances
  • All AWS resources are provisioned and managed using Terraform
  • Terraform allows for infrastructure visualization using its graph feature

AWS Lambda Function

I have created a simple AWS Lambda function using Python. This Lambda function is designed to receive two numbers as input from the user. The function takes these input numbers, calculates their sum, and generates a result.

AWS SNS Integration

To enable communication and notification, I have also utilized AWS SNS (Simple Notification Service). I used Terraform, an infrastructure-as-code tool, to create an SNS topic. The SNS topic serves as a communication channel for the Lambda function.

Publishing the Result

Once the Lambda function has calculated the sum, it publishes the result to the SNS topic. This allows subscribers or other components of your system to receive notifications or consume the result. The use of Terraform ensures that the SNS topic is provisioned automatically and consistently as part of your infrastructure setup.
